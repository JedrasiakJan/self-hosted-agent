# Self-Hosted-Agent: An Open-Source Sandboxed AI Coding Assistant

An open-source, highly adaptable AI engineering agent built from scratch using a custom, stateful **ReAct (Reasoning + Acting) execution loop**. This system is designed as a modular playground for developers who want to give local Large Language Models the power to navigate local workspaces, analyze code architecture, patch source scripts, and verify changes via standalone test suites using native subprocesses.

Originally conceptualized for cloud-managed commercial endpoints, the entire core has been fully refactored to focus on **Self-Hosted Open-Source LLMs (via LM Studio / OpenAI-compatible local servers)**. This shift guarantees absolute data privacy, zero data leakage, and zero dependency on restrictive subscription plans or cloud quotas. Because letting an LLM execute actions on a local machine is inherently risky, this toolkit features a transparent, code-level **Multi-Layered Security Sandbox**.

---

## 🛡️ Transparent Sandbox Architecture

In the spirit of open-source clarity, security should never be a black box. This project actively guards against prompt injection and model hallucinations via two structural, lightweight defensive layers:

### 1. Hardened Path Traversal Defense (`os.path.commonpath`)
Every single operational module interfacing with the disk (`get_files_info`, `get_file_content`, `write_file`, `run_python_file`) passes through a strict containment engine. By programmatically computing the longest shared absolute path between the sandbox root and the target location, the system locks the agent inside a hard-fenced micro-workspace (`./calculator`). Any deceptive attempt by the LLM to escape using directory traversal shortcuts (e.g., `../.env` or `../../etc/passwd`) is caught and blocked safely by Python before the OS ever processes the request.

### 2. Static Code Analysis Sandbox (AST Interception)
Before any source code or patch file generated by the LLM touches your filesystem or gets executed, the raw payload string is parsed into an **Abstract Syntax Tree (AST)** using Python's standard `ast` library. The validation routine recursively inspects the tree's nodes in flight. It instantly drops the execution or writing of any payload that tries to introduce unapproved, risky modules (`os`, `subprocess`, `sys`, `shutil`, `requests`) or dangerous built-in operations (`exec()`, `eval()`, `open()`, `compile()`).

## 🚧 Engineering Challenges & Migration Insights

Developing this project involved overcoming several complex infrastructure and cross-platform compatibility hurdles:

### 1. The WSL-to-Windows Network Boundary (The 429 to 403 / Connection Loop)
* **The Problem**: Initial deployment inside **WSL (Windows Subsystem for Linux)** using the free-tier Gemini API suffered from extreme rate-limiting (`429 Resource Exhausted`). Attempting to migrate to a self-hosted instance running **LM Studio on the Windows Host** created a network disconnection boundary. For the agent inside WSL, `localhost` points to the virtual Linux instance, making the host Windows server invisible. Furthermore, setting arbitrary virtual IP mappings often triggered system firewall rules (`Connection Timed Out`).
* **The Solution**: The workspace and agent orchestration logic were successfully ported directly onto the Windows environment via PowerShell. By unifying the execution context of both the Python script and the LM Studio API server onto the same host loop, seamless local networking via `localhost:1234` was established without risking security exposures.

### 2. Overcoming Strict Jinja Prompt Templates (`400 Bad Request`)
* **The Problem**: Most state-of-the-art open-source local models (e.g., *Qwen2.5-Coder*, *Llama-3.1*) utilize highly restrictive chat formatting templates wrapped in **Jinja code** inside LM Studio. When attempting to pass native `tools` parameters via standard API calls, the server crashed with a `400 Bad Request: Only user and assistant roles are supported!`. The prompt template completely rejected native structural `"tool"` roles and structured metadata.
* **The Solution**: Designed an innovative, template-agnostic **JSON Mode Enforcer**. Rather than relying on native API tool bindings, the entire tool catalog configuration is injected as raw text inside a unified `user` role prompt. The model is strictly instructed to respond *exclusively* in a rigid JSON structure containing `"thought"` and `"tool_call"` keys. The agent script uses custom regex parsers to intercept these JSON blocks in flight, execute the matching Python code locally, and append results back into the history stream seamlessly using safe `user`/`assistant` dialogue iterations.

## 🔧 Getting Started

### Prerequisites
* [Project UV](https://github.com) installed (modern, ultra-fast Python package manager) [–].
* [LM Studio](https://lmstudio.ai) running locally with an instruction/coding model loaded (*Qwen2.5-Coder-7B* or *Llama-3.1-8B* recommended).
* Ensure the LM Studio Local Server is **Started** on port `1234`.

### Installation & Run
1. Clone the repository and navigate to the project root directory [–].
2. Synchronize the environment and fetch Windows-specific lockfiles automatically:
   ```powershell
   uv sync
   ```
3. Boot up the agent loop by feeding it a task:
   ```powershell
   uv run main.py "Analyze the workspace and fix any bugs in calculator.py" --verbose
   ```
